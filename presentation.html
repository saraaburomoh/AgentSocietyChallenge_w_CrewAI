Armored Optimizations

⚡ Sequential over Hierarchical

Eliminates manager LLM overhead. No API token doubling. Deterministic execution with zero race conditions.

🍃 Token Diet

Strips "friends" field (5,000+ entries). Limits to 15 reviews per query. Truncates each review to 500 chars. Prevents ContextWindowExceeded crashes.

🛡️ Rate Limit Guard

Hard-capped at max_rpm=10. Single-threaded execution (max_workers=1). Prevents HTTP 429 errors on NVIDIA NIM free tier. Self-heals by pausing when limit is hit.

🔧 4-Layer JSON Sanitizer

Strategy 1: Find {"stars","review"} pattern. Strategy 2: Find {"predicted_rating","generated_review"}. Strategy 3: Direct json.loads(). Strategy 4: Regex star extraction fallback.

Evaluation Strategy: Embracing the "OR" Logic

📂 The Missing Entity Problem

  • In the real world, users visit new restaurants, and restaurants get new users.
  • The Challenge: The test set deliberately provides tasks where the User ID or Item ID is entirely missing from the local database.
  • The Pitfall: Traditional agents crash when a tool returns "No Data Found" because they expect complete information.

🧪 Verified Testing Environment

  • Implemented a robust Interaction Tool Wrapper that catches empty retrievals.
  • The LLM is explicitly instructed: "Proceed immediately using the information you already have."
  • Outcome: The pipeline successfully handles the 183 Challenge Tasks without a single crash, relying on contextual inference when data is missing.

Instead of failing on missing data, the pipeline adapts and infers — perfectly aligning with the core task design.

Challenge 1: RAG Tool Returned Wrong Data

Early Labs (Lab 1–5) — Configuration A: JSONSearchTool + Ollama (qwen2.5:7b)

❌ What Went Wrong

  • Used JSONSearchTool (semantic vector search) to look up users by ID
  • Query: user_id: nnImk681KaRqUVHlSfZjGQ
  • Got back: Michelle and Bill — completely wrong users
  • Agent retried 3+ times, always same wrong results
  • CrewAI printed: "I tried reusing the same input, I must stop"
  • Ollama 7B model then hallucinated a business called "The Love" — never retrieved, entirely fabricated from training data
  • Final prediction was 100% invalid — wrong user, wrong business

✅ Root Cause & Fix

  • Root cause: Yelp IDs are random base64 strings — they have NO semantic meaning in vector space
  • ChromaDB finds "nearest neighbor" vectors, which are just random unrelated records
  • Fix: Replaced JSONSearchTool with custom @tool functions using exact == string matching
  • No more wrong records. No more hallucinations.

Challenge 2: Token Limits & JSON Parsing Crashes

Labs 5–8 — Configuration B: JSONSearchTool + Groq (llama-3.1-8b-instant)

❌ Error B-1: HTTP 413 Rate Limit

openai.APIStatusError: Error 413
Limit 6000 TPM, Requested 6924
→ Request too large for model
  llama-3.1-8b-instant
  • Groq free tier: hard 6,000 TPM limit
  • Task 1 output + Task 2 JSON + system prompt = 6,924 tokens
  • 924 tokens over limit — entire request rejected
  • Waiting and retrying would NOT help — structurally too large

❌ Error B-3: JSONDecodeError

json.JSONDecodeError:
Expecting property name enclosed
in double quotes: line 1 col 114
  • review_subset.json is JSONL format
  • Review text contained literal \n newlines
  • Line-by-line parser broke mid-object
  • Fix: Used json.JSONDecoder().raw_decode() with streaming position-based parsing
  • Fix for 413: Token Diet — trim reviews to 300 chars, strip friends field, switch to llama-3.3-70b (higher quota)

Challenge 3: Context Window & Rate Limit Crashes

Labs 9–10 — Integration with Official Simulator

❌ OSError 1455 — Memory

Windows memory allocation error when loading HuggingFace embedding model for RAG. Caused by large model loading inside multi-threaded environment. Fixed by switching to CacheInteractionTool and avoiding local embedding entirely.

❌ Pydantic Crash

CrewAI passed a structured object (not a string) to the get_background_knowledge tool. Pydantic v2 rejected it. Fixed by changing the type hint from str to Any — one character fix, hours of debugging.

❌ LiteLLM Callback Error

list.remove(x): x not in list crash from LiteLLM's internal callback system during parallel crew runs. Fixed by adding litellm.callbacks = [] at flow startup — clears the corrupt callback list.

Challenge 4: Hierarchical vs Sequential

Lab 10 — The biggest architectural decision of the project

❌ Hierarchical Crew — What Went Wrong

  • Hierarchical mode requires a Manager Agent to delegate tasks
  • The Manager makes its own LLM call before every agent → doubles API usage
  • Caused constant HTTP 429 Rate Limit errors on NVIDIA NIM free tier
  • Manager sometimes delegated to the wrong agent or skipped tool use entirely

✅ Sequential Crew — The Fix

  • No manager overhead — tasks run one by one in strict order
  • Each agent passes its full output to the next as context
  • Halved API token consumption → no more 429 errors
  • Fully deterministic — same input always produces same agent behavior
  • Fixed the import: from src.crews.simulation_crew import SimulationCrew
  • Result: 100% stability across all 41 tasks, 0 crashes

Sequential process = the correct architecture for rate-limited, single-threaded API environments
